## Overview

This fork records a narrow documentation workflow: propose a useful README change, review it through a pull request, merge it, and inspect the resulting public history. Keeping the repository documentation-only makes every change small enough to understand without installing tools or running an application.

The 2026-07-18 audit found exactly four Markdown files and no source code, package manifest, asset directory, workflow, or executable entry point.

## Learning goals

- Practice branches, pull requests, reviews, and merges with a change that has real content.
- Learn to distinguish a reviewable contribution from an empty commit made only to create activity.
- Verify collaboration through the pull request, review, and commit history rather than through screenshots or claims.
- Keep attribution intact when adapting a fork.
- Understand that a GitHub action can be valid collaboration without guaranteeing an account Achievement.

## Recommended exercise

1. Create a branch with one independently useful documentation change.
2. Open a pull request that explains the reason and the expected rendered result.
3. Review the Markdown diff, links, language parity, and GitHub rendering from another account when appropriate.
4. Address review feedback, then merge through the pull request interface.
5. Use the resulting pull request and commit history as the evidence record.

This sequence is guidance for future practice. It does not claim that a specific pull request, review, merge, or Achievement occurred unless the public GitHub history proves it.

## Collaboration rules

- Use pull requests for shared README changes.
- Keep examples tied to actions that actually occurred on GitHub.
- Prefer one clear documentation concern per pull request.
- Never create empty commits, meaningless edits, fake reviews, or misleading activity.
- Do not present a badge, contribution event, or merged pull request as proof that GitHub awarded an Achievement.
- Keep the English, Simplified Chinese, and Japanese READMEs structurally and factually aligned.

## Verification checklist

Before merging a documentation exercise, verify:

- the change is useful when read without private context;
- the rendered Markdown has correct headings, tables, links, and image alt text;
- all three language selectors open the intended local README;
- the three translations preserve the same paths, links, facts, and limitations;
- the pull request author, reviewer actions, merge, and final commit are visible in public history when they are cited;
- no token, email secret, private URL, or unrelated personal data appears in the diff.

No automated test, lint, format, build, or CI command was found. Review the rendered README and public history manually.
